# Roadmap

## Where we are: May 2026

The first four components are available as a pre-pilot alpha in May 2026 ahead of the [UNICEF Climate & Health 2026](https://www.unicef.org/innovation/call-for-application-climate-and-health-2026) application. This is enough to show the product thesis in code and in a demo, but it is not the full pediatric module yet. The fifth component (`imci-rules`) ships after the grant decision.

## Components

| Package | What it does | State | Target |
|---|---|---|---|
| `growth-curves` | Compact offline WHO LMS z-score percentile calculator with interpolation | Pre-pilot alpha | Expand/reference validation after grant |
| `triage-ranges` | Pediatric vital sign reference ranges by age band | Pre-pilot alpha | Clinical reviewer pass before field pilot |
| `anomaly-detection` | Weekly z-score on rolling baseline per zone, for climate-sensitive pediatric indicators | Pre-pilot alpha | Calibration with real aggregate data after grant |
| `vaccinations` | Per-country vaccination schedule status logic | Pre-pilot alpha | MPPS/PAI validation before deployment |
| `imci-rules` | WHO IMCI deterministic danger sign rules | Spec stage | After grant decision |

The alpha APIs are documented in each package README. Component-level clinical specs live under `docs/components/` as they are reviewed.

A static demo web (`demo/`) consumes JSON generated by these packages to show the intended flow. The Local Node MVP (`apps/local-node/`) turns that flow into a small product surface: local intake, deterministic assessment, SQLite encounter storage, weekly aggregate surveillance input, weekly climate context and aggregate zone export.

## Local Node MVP

The Local Node is the current pre-pilot product target. It is intentionally small: FastAPI, SQLite and a bilingual browser UI. It proves the privacy boundary in code by keeping encounter-level data local, separating weekly surveillance input from patient intake, and exposing only aggregate zone-level signal JSON for export.

This is still not production clinical software. Production sync, user roles, full IMCI coverage, institutional adapters and field validation remain grant-funded work.

## If UNICEF Funding Lands

- **Months 1-3** — Harden the Local Node, finish IMCI rule coverage, integrate the open-source modules with the KYNODE edge node, and test internally on the existing Táchira node.
- **Months 4-6** — Build the clinical UI, review schedule/range assumptions with clinical and institutional partners, and prepare training materials for community health workers.
- **Months 7-9** — Pilot in selected medical centers in Venezuela, calibrate aggregate thresholds with real data, capture UX feedback and iterate.
- **Months 10-12** — Add institutional export adapters, publish a stronger public release, and prepare the path for broader deployment with partners.

## If UNICEF Funding Does Not Land

We still build the module. It just takes longer because we juggle it with paying work. Estimated extension: roughly 12 to 18 months for the same scope, depending on what other grants come through.

## What we will not do

- We will not turn this module into a diagnostic engine. It flags. The clinician decides. That line stays.
- We will not embed cloud dependencies in the local components. They must run in a clinic with no internet for weeks. That is the whole point of the design.
- We will not break the public API after v1.0 without a migration path. NGOs and ministries will write integrations against this; stability matters more here than novelty.
- We will not claim formal pilot validation before a formal pilot, and we will not describe the alpha as deployable end to end.
- We will not claim acute malnutrition status from weight-for-age alone. The current growth package provides growth-status flags only.

## What remains grant-funded work

- Complete WHO IMCI rule coverage.
- Complete integration with the KYNODE edge platform.
- Production clinical UI.
- Calibration with real aggregate data.
- Field pilots in medical centers.
- Community health worker training materials.
- Institutional export adapters.
- Formal validation with authorities and partners.

## How decisions get made

For now, by the small team that started this. As contributors come in, we'll move toward a lightweight RFC process documented in [CONTRIBUTING.md](CONTRIBUTING.md). For anything that touches clinical logic — triage thresholds, IMCI rules, vaccination schedules — we want at least one clinical reviewer on the PR before it ships.
