# Optional · Local LLM brief layer with Ollama

KYNODE Pediatric ships with a deterministic, rule-based AI surveillance brief that runs fully offline and requires zero configuration. This document explains how to optionally enable a **richer brief** generated by a **local large language model**, without breaking the offline-first promise of the product.

> **Why Ollama and not Anthropic / OpenAI?**
> The product's contract is that every layer of intelligence — clinical calculation, anonymisation, AI interpretation — runs at the edge, without sending data to a hosted SaaS API. Ollama runs the model on the same machine as the local node (or on a small server inside the clinic LAN), so the brief layer keeps the same offline guarantees as the rest of the system.

---

## What you get when you enable it

| Feature | Deterministic (default) | Local LLM (Ollama) |
|---|---|---|
| Internet required | ❌ No | ❌ No |
| Patient data sent anywhere | ❌ No | ❌ No |
| Per-call cost | $0 | $0 |
| Tone | Rule-based, repetitive | Naturally varied |
| Operational considerations | Curated 2-4 bullets per indicator | Bullets phrased to the actual numbers |
| Headline phrasing | Fixed template | Generated for the specific signal |
| Auditable | ✅ rule code is in `brief.py` | ⚠️ output depends on model weights |

The default deterministic generator is **always available as a fallback**. If Ollama is unreachable, the model is missing, or the response is malformed, the system silently falls back to the deterministic brief.

---

## Privacy contract (unchanged)

Whether you use the deterministic generator or the LLM, the brief sees only the **already-anonymised weekly aggregate export**. The chokepoint is the `safe_payload()` function in `apps/local-node/src/kynode_pediatric_local_node/brief.py`, which copies only allowlisted fields:

```
zone, week, indicator, count, baseline_mean, baseline_std,
z_score, flag, severity, signal_source, climate_context (without notes),
quality_warnings, contains_phi
```

A test (`test_brief_module_has_no_phi_field_in_safe_export_allowlist`) keeps this allowlist disjoint from any field that could contain PHI, even if the storage layer drifts.

When you point the local node at a remote Ollama instance inside your clinic LAN, the same allowlisted payload is the only data that travels — there is no patient identifier in the request body.

## Installation

### 1 · Install Ollama on the local node

Ollama is a single binary that runs on Linux, macOS and Windows. The installer takes a few minutes.

**Linux** (the typical clinic-node target):

```bash
curl -fsSL https://ollama.com/install.sh | sh
```

**macOS** (developer machines / pilot demos):

Download the installer from [ollama.com/download](https://ollama.com/download) and run it.

**Windows** (Surface tablets used as portable clinic nodes):

Download the Windows installer from the same link.

After install, Ollama runs as a background service and listens on `127.0.0.1:11434` by default. The service starts automatically on boot.

### 2 · Pull a model

Models are downloaded once and stored locally (typical sizes: 1.5GB – 7GB). For pediatric clinic deployments we recommend a small, fast, multilingual model that fits the resource budget of a Mini PC:

```bash
ollama pull llama3.2
```

Other tested options (pick one):

| Model | Size | Best for |
|---|---|---|
| `llama3.2` | 2GB | **Recommended default** · solid quality on EN/ES, fast on Mini PC |
| `qwen2.5:3b` | 1.9GB | Slightly better Spanish, similar speed |
| `phi3:mini` | 2.3GB | Microsoft's small instruct model, good fallback |
| `mistral:7b` | 4.1GB | Higher quality, needs ≥8GB RAM |

You only need one model installed at a time. The brief uses the same model for both English and Spanish; the language is controlled by the system prompt the local node sends.

### 3 · Verify Ollama is responding

```bash
curl http://localhost:11434/api/tags
```

You should see a JSON list including the model you pulled. If you get a connection error, start the service manually:

```bash
ollama serve
```

## Connect KYNODE Pediatric to Ollama

The local node reads three environment variables. Set them where you start the local node (systemd unit, Docker compose file, `.env` file loaded by your launcher, etc.):

```bash
export KYNODE_AI_BRIEF_PROVIDER=ollama
export KYNODE_AI_BRIEF_MODEL=llama3.2          # optional · defaults to llama3.2
export KYNODE_AI_BRIEF_ENDPOINT=http://localhost:11434/api/chat   # optional · default
```

Restart the local node. From now on, the **AI surveillance brief** button on the Surveillance page calls Ollama. The chip on the brief output reads **"Local LLM brief · Ollama (offline, post-anonymisation)"** instead of the deterministic chip.

### LAN deployment · one Ollama for many clinics

If you deploy several KYNODE Pediatric nodes in the same building or campus, you can run **one Ollama instance** on a slightly larger machine and point every clinic node at it:

```bash
# On the Ollama host (e.g. 10.0.0.10):
OLLAMA_HOST=0.0.0.0 ollama serve

# On each clinic node:
export KYNODE_AI_BRIEF_ENDPOINT=http://10.0.0.10:11434/api/chat
```

This is still offline — the LAN is the trust boundary. Make sure the Ollama host is not exposed beyond the clinic network (firewall rule, VPN-only access, etc.).

## Operational notes

- **First call after a cold start is slow** (5-10 seconds while the model loads into RAM). Subsequent calls are typically 1-3 seconds.
- **Ollama warms the model in memory** for ~5 minutes of idle, then unloads it. This is fine for clinic usage where briefs are generated once a week.
- **No usage logging is sent anywhere by Ollama**. If you want to audit prompts and responses for review, enable Ollama's local logging via `OLLAMA_DEBUG=1`.
- **Disable mid-shift if needed**: just unset `KYNODE_AI_BRIEF_PROVIDER` and restart the local node — the deterministic generator takes over without UI changes.

## Troubleshooting

| Symptom | Likely cause | Fix |
|---|---|---|
| Brief always shows "Deterministic template" chip | `KYNODE_AI_BRIEF_PROVIDER` not set or set to a value other than `ollama` | Check the env var; it must equal `ollama` (case-insensitive) |
| Brief panel shows "Local LLM request failed" toast | Ollama service not running, or wrong endpoint | `ollama serve`, then check `curl http://localhost:11434/api/tags` |
| Brief generation is slow on first call | Model is being loaded into RAM | Normal — subsequent calls are 1-3s |
| Brief content is in the wrong language | Model is too small or doesn't speak your language well | Try `qwen2.5:3b` for Spanish, or use the deterministic generator |
| Hosted reviewer URL still shows deterministic chip | The hosted public demo intentionally does NOT enable Ollama (LLM responses cost time/RAM and aren't needed for review). Run locally to see the LLM path. | Expected behaviour |

## Filesystem footprint

| Component | Disk |
|---|---|
| Ollama binary | ~150 MB |
| One model (llama3.2) | ~2 GB |
| Model cache, logs | ~50 MB |
| **Total** | **~2.2 GB** |

This is well within the disk budget of the Mini PC reference deployment (256 GB SSD) and leaves room for a second model if you want to A/B test.

## Removing Ollama

If a clinic decides not to use the local LLM after all:

```bash
# Stop and uninstall on Linux:
sudo systemctl stop ollama
sudo rm /usr/local/bin/ollama
rm -rf ~/.ollama   # deletes downloaded models

# Then unset the env var on the local node:
unset KYNODE_AI_BRIEF_PROVIDER
```

Restart the local node. The brief layer transparently returns to the deterministic generator. No change to the schema, the UI, or the audit trail.
